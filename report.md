# Отчёт по проекту Cyton

## 1. Введение

```text
Исходный код
    ↓
Лексический анализ
    ↓
Поток токенов
    ↓
Синтаксический анализ
    ↓
AST
    ↓
Семантический анализ
    ↓
Выполнение программы
```

В текущей версии проект реализует front-end языка и выполнение программ через **tree-walk interpreter**. Генерация x86-64 или LLVM IR пока не реализована.

---

## 2. Общая архитектура проекта

Проект построен как последовательный pipeline из независимых фаз:

```text
Файл .cy
    ↓
Lexer
    ↓
std::vector<Token>
    ↓
Parser
    ↓
Program / AST
    ↓
SemanticAnalyzer
    ↓
Interpreter
```

Каждая фаза отвечает за свою часть работы:

| Фаза             | Назначение                                                   |
| ---------------- | ------------------------------------------------------------ |
| Lexer            | Преобразует исходный текст в поток токенов                   |
| Parser           | Преобразует поток токенов в AST                              |
| AST              | Представляет структуру программы в виде дерева               |
| SemanticAnalyzer | Проверяет смысловую корректность программы                   |
| Interpreter      | Выполняет AST напрямую                                       |
| CLI              | Обрабатывает аргументы командной строки и запускает pipeline |

Такое разделение соответствует классической архитектуре компилятора, где каждая стадия получает результат предыдущей стадии и не смешивает свою ответственность с другими фазами.

## 4. Лексический анализатор

Лексический анализатор преобразует исходный текст программы в поток токенов.

Например, программа:

```cyton
var x = 10
print(x)
```

превращается в последовательность токенов:

```text
KwVar
Identifier "x"
Assign
IntLiteral "10"
Newline
Identifier "print"
LParen
Identifier "x"
RParen
```

Каждый токен хранит:

* тип токена;
* лексему;
* позицию в исходном файле: строку и столбец.

Позиция используется для вывода диагностических сообщений об ошибках.

---

### 4.1. Поддерживаемые токены

Лексер распознаёт:

* ключевые слова;
* идентификаторы;
* целочисленные литералы;
* вещественные литералы;
* строковые литералы;
* булевы значения;
* операторы;
* разделители;
* переносы строк;
* комментарии.

Примеры ключевых слов:

```cyton
fun
let
var
if
else
while
break
continue
return
true
false
struct
type
namespace
as
```

Примеры операторов:

```cyton
+
-
*
/
%
==
!=
<
<=
>
>=
=
&
|
^
~
<<
>>
|>
```

---

### 4.2. Комментарии

Поддерживаются однострочные комментарии:

```cyton
# comment
```

Также поддерживаются блочные комментарии:

```cyton
/* comment */
```

Блочные комментарии могут быть вложенными:

```cyton
/* outer /* inner */ outer */
```

Комментарии не попадают в поток токенов и не влияют на семантику программы.

## 5. Синтаксический анализатор

Синтаксический анализатор получает поток токенов и строит AST.

Парсер реализован методом рекурсивного спуска.

Этот метод был выбран по следующим причинам:

* он прост в реализации и отладке;
* каждая синтаксическая конструкция может быть описана отдельной функцией;
* удобно контролировать приоритет операторов;
* удобно выдавать понятные сообщения об ошибках.

---

### 5.1. Верхний уровень программы

На верхнем уровне программа может содержать:

* объявления функций;
* объявления структур;
* объявления синонимов типов;
* объявления пространств имён;
* инструкции верхнего уровня.

Пример:

```cyton
fun square(int64 x) -> int64:
    return x * x;

let value = square(5)
print(value)
```

В текущей версии инструкции верхнего уровня выполняются как скрытая точка входа программы — **hidden main**.

---

### 5.2. Разбор выражений

Выражения разбираются по уровням приоритета.

Например выражение:

```cyton
a + b * c
```

парсится как:

```cyton
a + (b * c)
```

а не как:

```cyton
(a + b) * c
```

Для этого в парсере используются отдельные функции для разных уровней приоритета:

```text
pipeline
cast
logical or
logical and
bitwise or
bitwise xor
bitwise and
equality
comparison
shift
term
factor
unary
postfix
primary
```

Такой подход позволяет явно задать приоритет и ассоциативность операторов.

## 6. AST

AST — это абстрактное синтаксическое дерево программы.

Исходный код после парсинга уже не хранится как обычный текст. Вместо этого программа представлена в виде набора узлов.

Например выражение:

```cyton
x + 2 * y
```

может быть представлено как дерево:

```text
Binary +
├── Identifier x
└── Binary *
    ├── Number 2
    └── Identifier y
```

AST нужен для дальнейших стадий:

* семантического анализа;
* интерпретации;
* потенциальной генерации кода.

---

### 6.1. Узлы выражений

В проекте реализованы следующие основные виды выражений:

| Узел            | Назначение             |
| --------------- | ---------------------- |
| `NumLitExpr`    | Целочисленный литерал  |
| `FloatLitExpr`  | Вещественный литерал   |
| `StringLitExpr` | Строковый литерал      |
| `BoolLitExpr`   | Булев литерал          |
| `IdentifExpr`   | Идентификатор          |
| `UnaryExpr`     | Унарная операция       |
| `BinaryExpr`    | Бинарная операция      |
| `CallExpr`      | Вызов функции          |
| `IndexExpr`     | Индексирование массива |
| `FieldExpr`     | Доступ к полю          |
| `CastExpr`      | Явное приведение типа  |
| `ArrayLitExpr`  | Литерал массива        |
| `StructLitExpr` | Литерал структуры      |

---

### 6.2. Узлы инструкций

Реализованы следующие основные виды инструкций:

| Узел            | Назначение                         |
| --------------- | ---------------------------------- |
| `LetInstr`      | Объявление неизменяемой переменной |
| `VarInstr`      | Объявление изменяемой переменной   |
| `ExprInstr`     | Выражение как инструкция           |
| `AssignInstr`   | Присваивание                       |
| `IfInstr`       | Условный оператор                  |
| `WhileInstr`    | Цикл                               |
| `BreakInstr`    | Выход из цикла                     |
| `ContinueInstr` | Переход к следующей итерации цикла |
| `ReturnInstr`   | Возврат из функции                 |
| `BlockInstr`    | Блок инструкций                    |

## 7. Семантический анализатор

Семантический анализатор проверяет программу после построения AST.

Синтаксически корректная программа может быть всё ещё неправильной по смыслу.

Например:

```cyton
print(x)
```

Если `x` не была объявлена ранее, то программа синтаксически корректна, но семантически ошибочна.

---

### 7.1. Задачи семантического анализа

Семантический анализатор выполняет:

* разрешение имён;
* проверку областей видимости;
* проверку повторных объявлений;
* проверку использования переменных до объявления;
* проверку мутабельности переменных;
* проверку типов;
* проверку вызовов функций;
* проверку количества и типов аргументов;
* проверку возвращаемых значений функций;
* проверку `break` и `continue`;
* проверку корректности доступа к полям структур.

---

### 7.2. Области видимости

Области видимости реализованы как стек таблиц символов.

При входе в новый блок создаётся новая область видимости.

При выходе из блока область удаляется.

Пример:

```cyton
let x = 10

if true:
    let y = 20
    print(x)
    print(y);
```

Внутри блока доступны и `x`, и `y`.

Но после выхода из блока `y` уже недоступна:

```cyton
print(y) # error
```

---

### 7.3. Shadowing

В языке допускается перекрытие имён во вложенных областях видимости.

Пример:

```cyton
let x = 1

if true:
    let x = 2
    print(x);
```

Внутри блока используется внутреннее объявление `x`.

---

### 7.4. `let` и `var`

В языке различаются неизменяемые и изменяемые переменные.

`let` создаёт неизменяемую переменную:

```cyton
let x = 10
```

После этого присваивание в `x` запрещено:

```cyton
x = 20 # error
```

`var` создаёт изменяемую переменную:

```cyton
var y = 10
y = 20
```

Семантический анализатор проверяет это правило до выполнения программы.

---

### 7.5. Проверка типов

В языке поддерживаются основные типы:

* `int64`;
* `uint64`;
* `float64`;
* `bool`;
* `string`;
* `unit`;
* структуры;
* массивы.

Пример корректного кода:

```cyton
let int64 x = 10
let string name = "Cyton"
let bool flag = true
```

Пример ошибки типов:

```cyton
let int64 x = "hello" # error
```

---

### 7.6. Функции

Семантический анализатор проверяет объявления и вызовы функций.

Пример функции:

```cyton
fun add(int64 a, int64 b) -> int64:
    return a + b;
```

Пример вызова:

```cyton
print(add(2, 3))
```

Проверяется:

* существует ли функция;
* совпадает ли количество аргументов;
* совместимы ли типы аргументов;
* возвращает ли функция значение нужного типа.

---

### 7.7. Проверка `return`

Для функций с возвращаемым типом, отличным от `unit`, проверяется наличие `return`.

Например:

```cyton
fun f(int64 x) -> int64:
    if x > 0:
        return 1;
```

Такая функция некорректна, потому что не во всех ветках возвращается значение.

Корректный вариант:

```cyton
fun f(int64 x) -> int64:
    if x > 0:
        return 1
    else:
        return 0;
    ;
```

---

### 7.8. `break` и `continue`

Инструкции `break` и `continue` разрешены только внутри цикла.

Пример корректного использования:

```cyton
while true:
    break;
```

Пример ошибки:

```cyton
break # error
```

Семантический анализатор отслеживает, находится ли текущая инструкция внутри цикла.

---

## 8. Интерпретатор

В текущей версии Cyton выполняет программу через **tree-walk interpreter**.

Это означает, что программа не переводится в машинный код. Вместо этого интерпретатор обходит AST и выполняет узлы напрямую.

Например:

```cyton
let x = 10
print(x + 5)
```

выполняется так:

1. Интерпретатор создаёт переменную `x`.
2. Вычисляет значение `10`.
3. Сохраняет значение в окружении.
4. Вычисляет выражение `x + 5`.
5. Вызывает встроенную функцию `print`.

---

### 8.1. Runtime-значения

Значения во время выполнения представлены через `std::variant`.

Поддерживаемые runtime-значения:

| Значение  | Назначение          |
| --------- | ------------------- |
| `unit`    | Отсутствие значения |
| `int64`   | Целое число         |
| `float64` | Вещественное число  |
| `bool`    | Булево значение     |
| `string`  | Строка              |
| `array`   | Массив              |
| `struct`  | Структура           |

Такой подход удобен для интерпретатора, потому что одно значение может хранить данные разных типов.

---

### 8.2. Окружение переменных

Интерпретатор хранит переменные в runtime-окружении.

Окружение также реализовано как стек областей видимости.

Когда выполняется блок, создаётся новая область.

Когда блок завершается, область удаляется.

---

### 8.3. Управление потоком

Для обработки `break`, `continue` и `return` используется специальный результат выполнения инструкции.

Инструкция может завершиться одним из вариантов:

* обычное продолжение выполнения;
* `break`;
* `continue`;
* `return`.

Это нужно, потому что `return` и `break` должны выходить из вложенных блоков.

Например:

```cyton
while true:
    if true:
        break;
    ;
;
```

`break` должен выйти не только из `if`, но и из цикла `while`.

---

### 8.4. Value semantics

В языке используется value semantics.

Это означает, что при присваивании или передаче значения копируются по значению.

Для простых типов это естественно:

```cyton
var a = 10
var b = a
b = 20

print(a) # 10
print(b) # 20
```

Для массивов и структур в интерпретаторе используется глубокое копирование, чтобы изменение копии не изменяло исходный объект.

## 11. Дополнительные возможности

Кроме базовых возможностей, в проекте реализованы дополнительные элементы:

### 11.1. Hidden main

В текущей версии инструкции верхнего уровня могут выполняться напрямую.

Например:

```cyton
print("Hello")
```

Такой код не требует явного объявления функции `main`.

Это удобно для небольших программ и тестирования языка.

---

### 11.2. REPL

REPL позволяет выполнять код интерактивно.

Запуск:

```bash
./build/cyton
```

или:

```bash
./build/cyton --repl
```

---

### 11.3. Отладочный вывод токенов

Флаг:

```bash
--dump-tokens
```

позволяет посмотреть результат работы лексера.

Пример:

```bash
./build/cyton example/hello.cy --dump-tokens
```

---

### 11.4. Отладочный вывод AST

Флаг:

```bash
--dump-ast
```

позволяет посмотреть AST после парсинга.

Пример:

```bash
./build/cyton example/hello.cy --dump-ast
```

---

### 11.5. Перегрузка функций

В проекте поддерживается несколько функций с одним именем, если они отличаются типами параметров.

Пример:

```cyton
fun show(int64 x) -> unit:
    print(x);

fun show(string x) -> unit:
    print(x);

show(10)
show("hello")
```

---

### 11.6. Пространства имён

Пространства имён позволяют группировать объявления.

Пример:

```cyton
namespace Math:
    fun square(int64 x) -> int64:
        return x * x;
;

print(Math::square(5))
```

## 15. Диагностика ошибок

Компилятор/интерпретатор выводит сообщения об ошибках при обнаружении некорректной программы.

Обрабатываются следующие виды ошибок:

* лексические ошибки;
* синтаксические ошибки;
* использование необъявленной переменной;
* повторное объявление имени;
* изменение переменной, объявленной через `let`;
* несовпадение типов;
* неправильный вызов функции;
* отсутствие возвращаемого значения;
* некорректное использование `break`;
* некорректное использование `continue`;
* деление на ноль;
* выход за границы массива.

Пример:

```cyton
let x = 10 / 0
```

Во время выполнения будет выведена runtime-ошибка.

## 16. Принятые проектные решения

### 16.1. Использование C++23

Проект реализован на C++23. Это позволяет использовать современные возможности языка C++ и стандартной библиотеки.

---

### 16.2. Использование C++20 modules

Часть интерфейсов оформлена через `.cppm`-модули.

Это позволяет отделить интерфейсы фаз компилятора от их реализаций и уменьшить зависимость от классических заголовочных файлов.

---

### 16.3. Recursive descent parser

Для парсера выбран метод рекурсивного спуска.

Преимущества этого подхода:

* простота реализации;
* понятная структура кода;
* удобная обработка ошибок;
* прямое соответствие между грамматикой и функциями парсера;
* возможность вручную контролировать приоритеты операторов.

---

### 16.4. AST через иерархию классов

AST реализован как набор структур и классов для разных видов выражений и инструкций.


---

### 16.5. Tree-walk interpreter

В текущей версии используется tree-walk interpreter.

Он был выбран как промежуточный этап разработки, потому что позволяет:

* быстрее проверить синтаксис языка;
* быстрее проверить семантику;
* тестировать программы без реализации машинного кода;
* проще отлаживать поведение языка;
